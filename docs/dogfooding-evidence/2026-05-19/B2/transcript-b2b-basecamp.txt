§4. WHAT IS NOT IN THE LOG (the silent-drop signal)
------------------------------------------------------------------------
Across all 350 captured lines:
  grep -niE 'variant|platform|main file|mainFilePath|known modules|
             unsupported|skipping|incompatible|not loadable|error|warn' \
       /tmp/b2-bad.log | grep -i delivery
produces NO output. PackageManagerLib accepted the user-modules
directory, enumerated delivery_module from manifest.json, and discarded
it from the load-eligible set (because manifest.json's
main."darwin-arm64-dev" key is not in platformVariantsToTry() for a
LGPM_PORTABLE_BUILD host) without emitting a single [info]/[warn]/[error]
line that mentions the variant mismatch, the unsupported platform key,
or any operator-actionable explanation. lgpm list against the same dir
happily reports the module as installed (it reads manifest.json without
filtering by variant — verified separately).

The only downstream signal — captured during this repo's M2 cross-node
proof against bin-macos-app before the local mitigation, cited verbatim
in scripts/basecamp-instance.sh line 217 and in delivery-dogfooding.md —
is the much-later runtime line:
  '[warning] Module not found in known modules: <name>'
  'Failed to load core dependency "<name>" for "<consumer>"'
which fires only when a dependent UI plugin or module asks the package_manager
to load the dropped module. By that point the operator has no breadcrumb
back to the variant-mismatch root cause.
